Step 1 · Parse document

The workflow starts from startExtractionEvent and calls parseDocument(filePath, mimeType) inside withRetry(...). Success emits documentParsedEvent.

Input / output

  • Consumes: startExtractionEvent
  • Calls: parseDocument
  • Emits: documentParsedEvent

Why it matters

Normalizes the source document into parsed text before chunking and extraction begin.

Step 2 · Chunk document and fan out tasks

When documentParsedEvent arrives, the workflow runs chunkDocument(fullText, documentId), computes expectedResultCount, resets collectedResults, and emits one extractionTaskEvent per chunk/type combination.

State mutation

  • state.expectedResultCount = chunks × (entityTypes + factTypes)
  • state.collectedResults = []

Emitted tasks

Each task contains chunk, typeName, typePrompt, and kind (entity or fact).

Step 3 · Per-type extraction

Every extractionTaskEvent routes into either extractEntityType or extractFactType. Even failures are converted into extractionResultEvent objects with an error field, so the workflow can continue.

Branches

  • kind === 'entity' → extractEntityType
  • kind === 'fact' → extractFactType

Failure strategy

Task failures do not crash the workflow immediately; they become structured result events and are handled later.

Step 4 · Fan-in collection via workflow state

Each extractionResultEvent is pushed into state.collectedResults. Once the number of received results reaches expectedResultCount, the workflow aggregates successful entities, successful facts, and failures into extractionCollectedEvent.

Collected outputs

  • entities[]
  • facts[]
  • failures[]

Key pattern

This is the fan-in mechanism: state accumulation instead of a dedicated collectEvents helper.

Step 5 · Deduplicate entities

This is the explicit deduplication handler. It consumes extractionCollectedEvent, runs deduplicateEntities(entities), and emits dedupCompleteEvent.

Exactly what is deduplicated?

  • Deduplicated: entities
  • Not deduplicated: facts

Outcome

The next step receives deduped entities, unchanged facts, and any failures collected earlier.

Step 6 · Persist results and optionally mark partial

The workflow persists results with persistResults(prisma, documentId, entities, facts) inside withRetry(...). If failures exist, it updates the document row to status: 'partial' with a summarized error string, then emits extractionCompleteEvent.

Persistence behavior

  • Successful partial output is still persisted.
  • Retries are used around persistence.

Completion

ExtractionWorkflow.run() waits on ctx.stream.untilEvent(extractionCompleteEvent) before returning.
